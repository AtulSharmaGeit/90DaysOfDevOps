## Create Our First Pod (Nginx)
### Create a file called `nginx-pod.yaml`:
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:
    app: nginx
spec:
  containers:
  - name: nginx
    image: nginx:latest
    ports:
    - containerPort: 80
```
Apply the manifest:
```bash
kubectl apply -f nginx-pod.yaml
```

## Create a Custom Pod (BusyBox)
Write a new manifest `busybox-pod.yaml` from scratch:
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: busybox-pod
  labels:
    app: busybox
    environment: dev
spec:
  containers:
  - name: busybox
    image: busybox:latest
    command: ["sh", "-c", "echo Hello from BusyBox && sleep 3600"]
```

Apply the manifest:
```bash
kubectl apply -f busybox-pod.yaml
```

Check pod status:
```bash
kubectl get pods
```
- Wait until `STATUS` shows **Running**.

Check logs:
```bash
kubectl logs busybox-pod
```
- We should see:
    
  ![image alt](https://github.com/atulsharmadevops/90DaysOfDevOps/blob/0558902c53265dc0d62396938db45eca5b9f945a/2026/day-51/Screenshots/Screenshot%20(614).png)

### Why the `command` field matters ?
 **Ans.** BusyBox has no long-lived default process like Nginx does. Without `sleep 3600`, the container exits immediately after printing the message. Kubernetes interprets the exit as a crash and restarts the container - repeatedly - resulting in `CrashLoopBackOff`.
 
Adding `sleep 3600` keeps the container alive long enough to inspect logs and exec into it.
 
| Without `sleep` | With `sleep 3600` |
|---|---|
| Container exits immediately | Container stays alive for 1 hour |
| Kubernetes restarts it in a loop | Pod stays in `Running` state |
| Status: `CrashLoopBackOff` | Status: `Running`  |

## Imperative vs Declarative
There are two ways to manage resources in Kubernetes:
 
| Approach | Method | Best for |
|---|---|---|
| **Declarative** | Write YAML → `kubectl apply` | Production, version-controlled configs |
| **Imperative** | `kubectl run`, `kubectl create` | Quick tests, scaffolding, exploration |

### Create a pod without a YAML file
```bash
kubectl run redis-pod --image=redis:latest
```

### Check it
```bash
kubectl get pods
```

![image alt](https://github.com/atulsharmadevops/90DaysOfDevOps/blob/0558902c53265dc0d62396938db45eca5b9f945a/2026/day-51/Screenshots/Screenshot%20(616).png)

Now extract the YAML that Kubernetes generated:
```bash
kubectl get pod redis-pod -o yaml
```

![image alt](https://github.com/atulsharmadevops/90DaysOfDevOps/blob/0558902c53265dc0d62396938db45eca5b9f945a/2026/day-51/Screenshots/Screenshot%20(617).png)

Comparing the generated YAML against our hand-written manifests reveals how much Kubernetes fills in automatically:
| Field | Hand-written manifest | Generated by Kubernetes |
|---|---|---|
| `apiVersion`, `kind`, `metadata.name` | ✓ Specified | ✓ Specified |
| `uid`, `resourceVersion`, `creationTimestamp` | ✗ Not in YAML | ✓ Added automatically |
| `restartPolicy: Always` | ✗ Not specified | ✓ Default applied |
| `dnsPolicy`, `terminationGracePeriodSeconds` | ✗ Not specified | ✓ Default applied |
| `status` block | ✗ Not in YAML | ✓ Live cluster state |
 
### Scaffold a manifest with `--dry-run`
Use `--dry-run=client` to generate a manifest without creating any resource - a fast way to bootstrap YAML that we can then customize:
```bash
kubectl run test-pod --image=nginx --dry-run=client -o yaml
```

Save the dry-run output to a file.

## Validate Before Applying
Always validate manifests before applying them to catch errors early.
### Check if the YAML is valid without actually creating the resource (syntax check only) - Client-Side Validation :
```bash
kubectl apply -f nginx-pod.yaml --dry-run=client
```

### Validate against the cluster's API (Server-Side Validation):
```bash
kubectl apply -f nginx-pod.yaml --dry-run=server
```

![image alt](https://github.com/atulsharmadevops/90DaysOfDevOps/blob/0558902c53265dc0d62396938db45eca5b9f945a/2026/day-51/Screenshots/Screenshot%20(620).png)

Now intentionally break YAML (remove the image field or add an invalid field) and run dry-run again. See what error we can get:
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: broken-nginx-pod
spec:
  containers:
  - name: nginx
    # image field removed
    ports:
    - containerPort: 80
```
Run validation again:
```bash
kubectl apply -f broken-nginx-pod.yaml --dry-run=client
kubectl apply -f broken-nginx-pod.yaml --dry-run=server
```
When the image field is missing, Kubernetes will return:

## Key Takeaway
 
When we delete a standalone Pod, it is gone permanently. No controller exists to recreate it. This is why production workloads use **Deployments** instead of bare Pods - Deployments maintain a desired replica count and automatically recreate pods if they are deleted or crash.
 
> Bare Pods are useful for learning and debugging. In production, always wrap our pods in a Deployment.
